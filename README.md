# System-analysis
system analysis class repo

The DriverPass project focused on designing a system for DriverPass, a company that helps students prepare for their driving tests. The client wanted a web-based system where customers could register for accounts, purchase training packages, take online practice tests, schedule driving lessons, make payments, and communicate with instructors. The system also needed administrative tools for managing users, instructors, appointments, and updates from the DMV.

One thing I think I did particularly well was translating the client's interview into clear system requirements and UML diagrams. Creating use case, activity, sequence, and class diagrams helped me understand how each part of the system worked together and how different users interacted with the system.

If I could revise one part of my work, I would spend more time refining the UML diagrams. While they met the project requirements, I would improve the layout, organization, and level of detail to make them easier for both developers and clients to understand.

To design the system, I first identified the client's goals and then determined the needs of each user, including customers, instructors, administrators, and IT staff. I used those requirements to build diagrams that showed how the system should function. Considering user needs is important because the software must solve real problems and be easy for people to use. A well-designed system should make tasks simpler instead of creating additional confusion.

In the future, I will continue to begin software design by gathering requirements from stakeholders before creating diagrams or planning implementation. I would also use UML diagrams early in the design process to visualize workflows and relationships before development begins. Breaking large systems into smaller pieces, reviewing designs with stakeholders, and making improvements based on feedback are strategies I will continue to use on future software projects.
